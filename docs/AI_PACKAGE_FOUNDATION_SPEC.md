# AI Package Foundation Spec

Status: Draft foundation spec  
Date: 2026-04-07  
Scope: Core DBVC planning baseline for AI-facing sample packages and AI submission package intake

## Purpose

This document defines the foundational contract for a new DBVC AI package workflow.

The workflow has two connected parts:

1. DBVC generates a site-specific sample package that users can upload to browser-based AI sessions such as ChatGPT or Claude.
2. The AI returns a stricter DBVC AI submission package that DBVC validates and translates before handing off to the plugin's existing import/upload flow.

This spec exists to lock the product boundary, archive/package model, canonical entity shape, and validation states before writing a detailed implementation guide.

## Problem Statement

DBVC can already export and import JSON payloads for posts, CPTs, and taxonomies, but that system is optimized for DBVC sync and proposal workflows rather than human-guided AI authoring sessions.

Users need a guided way to:

- download current site structure and field expectations
- give that structure to an AI system
- receive back DBVC-compatible payloads
- validate those payloads safely
- import them without rewriting the legacy DBVC importer

The new AI package feature should improve AI usability without weakening DBVC's current import guarantees.

## Goals

- Add a new `Tools` submenu under the DBVC top-level admin menu.
- Add a core DBVC tool named `Download Sample Entities`.
- Generate site-specific sample packages for selected CPTs and taxonomies.
- Support two shape modes:
  - `conservative`
  - `observed_shape`
- Default to `conservative`.
- Support configurable value styles:
  - `blank`
  - `dummy`
- Support configurable sample variants:
  - `single`
  - semantic multi-variant sets such as `minimal`, `typical`, and `maximal`
- Export AI-readable documentation at the package root and beside each sample entity.
- Allow user-authored notes and rules to be included in generated packages.
- Add a stricter AI submission package intake layer in front of the current DBVC import/upload path.
- Keep current DBVC upload/import behavior intact for legacy JSON and ZIP workflows.
- Keep import deterministic. The importer must not call external AI services at runtime.

## Non-Goals

- Replace DBVC proposal bundles or proposal review workflows.
- Replace generic sync-folder ZIP/JSON uploads.
- Make outbound AI credentials mandatory for this feature.
- Perform AI-driven repair or rewriting during import.
- Ship first-release support for menus, options, attachments, or media packaging as part of the AI submission contract.
- Couple this feature to the Content Migration addon runtime.

## Product Boundary

This feature introduces a new AI-focused layer while preserving DBVC's current importer.

The system should be understood as three layers:

1. `Sample package generation`
   - creates AI-facing, site-specific reference material
   - outputs canonical import-oriented templates plus docs
2. `AI submission package intake`
   - detects and validates a strict AI package contract
   - produces a validation report
   - translates accepted payloads into standard DBVC sync-compatible JSON
3. `Existing DBVC importer`
   - remains the system of record for actual content creation and updates
   - continues to process standard DBVC JSON after intake translation

This means the new feature is additive. It does not replace the current importer contract.

## Core Concepts

### Sample Entities Package

A ZIP file generated by DBVC for users to hand to an AI session. It contains:

- canonical DBVC-shaped sample entity JSON templates
- schema inventory artifacts
- validation rules and field catalogs
- top-level AI guidance docs
- per-sample markdown instructions

This package is AI context only. It is not directly importable.

### AI Submission Package

A stricter ZIP file produced by an AI system or by a human following the sample package contract. It contains:

- a required package manifest
- entity JSON payloads in a strict archive layout
- optional notes/report docs

This package is import-adjacent, not proposal-based. It must pass AI-package intake validation before DBVC routes its contents into the existing import flow.

### AI Package Intake Layer

A new validation and translation layer that runs before the existing importer. It should:

- detect AI package manifests
- validate archive structure and contents
- reject blocked packages without mutating the sync folder
- translate valid packages into standard DBVC sync JSON
- then hand off to the existing DBVC importer

### Canonical Template JSON

The JSON templates in the sample package should mirror the DBVC importer contract closely enough that the AI can fill them in and produce importable output.

These templates should not be raw live exports. They should be cleaned, import-oriented templates that:

- preserve the actual DBVC field shape
- preserve real object keys and field names
- omit or neutralize runtime-only noise
- use blank or dummy values instead of real content

## Architecture Principles

- Core feature first. No addon dependency is required for sample package generation or AI package intake.
- Reuse existing DBVC export/import payload patterns where they are already correct.
- Prefer exact importer-compatible JSON over abstract schema-only representations.
- Add AI-oriented docs around the canonical JSON rather than inventing a separate data model the AI must translate back from.
- Use deterministic generation. The same settings should produce the same package structure and variant names.
- Keep validation strict enough to protect import integrity, but configurable enough to support warning-based review flows.

- `Tools`

- `Download Sample Entities`

## Package Types

Two package types are defined for this workflow.

### `dbvc_ai_sample_package`

Produced by DBVC. Intended to be uploaded into AI sessions.

### `dbvc_ai_submission_package`

Produced by AI or by a human following the sample package contract. Intended to be uploaded into DBVC.

## Manifest Strategy

Both package types should use the same root manifest filename:

- `dbvc-ai-manifest.json`

The archive root should contain the manifest directly. Intake may tolerate a single wrapper directory for compatibility with common ZIP tools, but the canonical layout is root-first.

Manifest versioning should be independent from DBVC proposal schema versions.

Recommended top-level manifest keys:

- `package_type`
- `package_schema_version`
- `generated_at`
- `dbvc_version`
- `site_origin`
- `site_fingerprint`
- `artifacts`

## Canonical Entity JSON Contract

### Guiding Rule

The canonical AI entity templates should mirror the DBVC importer contract, not a separate abstract schema contract.

That means:

- the AI sees the same top-level field families that DBVC imports
- the AI fills out a known DBVC-compatible shape
- the intake layer validates and translates the package without needing to reinterpret a custom DSL

### Post and CPT Entity Contract

The canonical post template should closely follow the JSON shape consumed by DBVC post import logic.

Required fields:

- `ID`
- `post_type`
- `post_title`
- `post_name`

Allowed optional fields:

- `post_status`
- `post_content`
- `post_excerpt`
- `post_date`
- `post_date_gmt`
- `post_parent`
- `menu_order`
- `post_author`
- `post_password`
- `comment_status`
- `ping_status`
- `meta`
- `tax_input`

Rules:

- `ID` must exist. Use `0` for net-new entities.
- `post_type` must match a supported post type on the current site.
- `post_name` must exist in AI submission packages so slug-based matching and reference resolution stay deterministic.
- `meta` must be an object keyed by meta key.
- `tax_input` must be an object keyed by taxonomy.
- `tax_input` values may use either:
  - a string slug
  - an object with `slug`, `name`, and optional `parent`

Example `tax_input` shape:

```json
{
  "service_type": [
    {
      "slug": "consulting",
      "name": "Consulting",
      "parent": null
    }
  ],
  "category": ["news"]
}
```

System-managed or discouraged fields:

- `vf_object_uid`
- `dbvc_post_history`
- `_dbvc_import_hash`
- proposal-specific reviewer or resolver data

These fields may appear in live DBVC exports, but they should not be part of the standard AI authoring contract for net-new content.

### Term Entity Contract

The canonical term template should closely follow the JSON shape consumed by DBVC term import logic.

Required fields:

- `term_id`
- `taxonomy`
- `name`
- `slug`

Allowed optional fields:

- `description`
- `parent`
- `parent_slug`
- `meta`

Rules:

- `term_id` must exist. Use `0` for net-new terms.
- `taxonomy` must match a registered taxonomy on the current site.
- `meta` must be an object keyed by meta key.
- parent linkage should prefer `parent_slug` for portability.

System-managed or discouraged fields:

- `vf_object_uid`
- `parent_uid`
- import/proposal history blobs

### Canonical Template Rules

Sample templates must:

- use the real post type and taxonomy keys from the site
- use the real meta key names and ACF field names discovered by the generator
- avoid real content values
- prefer empty, null, or dummy placeholders depending on value style
- preserve field shapes that the importer expects
- expose logical ACF field names rather than raw underscore reference meta keys
- rely on the intake layer to synthesize ACF reference meta where required

## Shape Modes

### `conservative`

The default mode.

Sources:

- post type registration and supports
- attached taxonomies
- taxonomy registration
- registered meta
- ACF field groups and fields when ACF is available
- known DBVC-safe core field families

This mode should avoid speculative field inclusion.

### `observed_shape`

An additive mode that augments conservative discovery with shape signals sampled from existing site content.

Observed mode should:

- inspect a bounded number of existing entities per object type
- discover recurring meta keys and field presence patterns
- record field provenance and frequency
- never export live values as sample content

Observed mode must still emit blank or dummy placeholders rather than copied site values.

## Value Styles

### `blank`

Fields are emitted with neutral empty values where possible:

- empty string
- `null`
- empty array
- `0` for neutral IDs in create templates

### `dummy`

Fields are emitted with safe synthetic placeholders:

- `"Sample Page Title"`
- `"sample-page-title"`
- `"Replace with final content."`

Dummy mode should never use real production content.

## Variant Sets

Variant counts should be semantic rather than arbitrary.

Recommended variant options:

- `single`
- `minimal`
- `typical`
- `maximal`

Semantics:

- `minimal` contains only required and most-likely core fields
- `typical` contains normal authoring fields and common tax/meta/ACF families
- `maximal` contains the broadest safe contract the generator can support for the selected object type

## Package Documentation Requirements

### Top-Level Docs

Every sample package should contain top-level docs intended for AI agents and human operators.

Required docs:

- `README.md`
- `AGENTS.md`
- `STARTER_PROMPT.md`
- `OUTPUT_CONTRACT.md`
- `USER_RULES.md`
- `VALIDATION_RULES.md`

Purpose:

- `README.md`: explain what the package is
- `AGENTS.md`: explain how an AI should use the package
- `STARTER_PROMPT.md`: provide copy-paste starter prompts
- `OUTPUT_CONTRACT.md`: define required returned archive structure
- `USER_RULES.md`: export user-authored notes and editorial rules
- `VALIDATION_RULES.md`: explain what DBVC will enforce before import

### Per-Sample Markdown

Each sample JSON should have a sibling `.md` file.

Each per-sample markdown file should include:

- background context about the object type
- content authoring guidance
- required vs optional fields
- taxonomy assignment guidance
- meta field context
- ACF field context
- available values for choice-based fields where known
- relationship expectations
- known exclusions
- provenance notes:
  - `registered`
  - `acf`
  - `observed`

## Structured Rule Export

User-authored notes should not be markdown-only if DBVC intends to validate against them later.

The sample package should include a machine-readable rules artifact:

- `schema/validation-rules.json`

This file should carry structured rules where available, such as:

- allowed statuses
- required taxonomies
- allowed taxonomy values when explicitly configured
- required meta keys
- allowed ACF choice values
- blocked fields

Markdown docs should explain the same rules in human-readable language, but machine enforcement should not depend on markdown parsing.

## AI Submission Intake Model

The AI submission package should be processed by a new intake layer before existing import routing.

Intake responsibilities:

- detect `dbvc-ai-manifest.json`
- identify package type
- validate manifest and archive layout
- validate each entity file
- build a validation report
- translate accepted entity JSON into standard DBVC sync-compatible files
- then hand off to the current DBVC import path

The intake layer must not call external AI services.

## Validation States Before Import

Validation should produce one package-level result:

- `valid`
- `valid_with_warnings`
- `blocked`

### `valid`

The package satisfies required structure and content rules and can be translated into standard DBVC sync JSON automatically.

### `valid_with_warnings`

The package is structurally acceptable, but non-blocking issues were detected.

Examples:

- optional fields omitted
- advisory user rules not followed
- extra doc files included
- unsupported optional metadata that can be ignored safely

This state should support a configurable handling policy:

- continue automatically
- continue only after explicit confirmation

### `blocked`

The package cannot be translated safely and must not mutate the sync folder or importer state.

Examples:

- missing or invalid manifest
- unsupported package type or schema version
- malformed JSON
- unknown or unsupported post type
- unknown taxonomy
- missing required fields
- duplicate entity collisions inside the package
- invalid field shape for canonical contract
- protected/system-managed fields used in a forbidden way
- machine-enforced validation rule violations

## Validation Report Model

The intake layer should emit a structured report that can be displayed in the UI and logged for debugging.

Recommended retained report files:

- `reports/validation-report.json`
- `reports/validation-summary.md`

The JSON report is the canonical machine-readable artifact. The markdown summary is optional convenience output for operators.

### Validation Entity Record Shape

Each `entities[]` record should describe one logical submitted entity.

Recommended fields:

- `entity_key`
- `source_path`
- `entity_kind`
- `object_type`
- `operation_intent`
- `match_basis`
- `resolved_target`
- `status`
- `issue_codes`

Notes:

- `entity_key` should be deterministic and human-readable. Recommended patterns:
  - `post:{post_type}:{slug}`
  - `term:{taxonomy}:{slug}`
- `match_basis` should record how an update-intent entity resolved:
  - `vf_object_uid`
  - `slug`
  - `id`
  - `create`
- `resolved_target` may be `null` for create-intent entities until import executes, but update-intent entities should have a resolved local target before translation proceeds.

### Validation Issue Record Shape

Each issue should include:

- `severity`
- `code`
- `scope`
- `source_path`
- `entity_key`
- `field_path`
- `message`
- `remediation_hint`
- `details`

Recommended severities:

- `warning`
- `error`

Recommended scopes:

- `package`
- `file`
- `entity`
- `field`

Recommended issue-code families:

- manifest and archive:
  - `manifest_missing`
  - `manifest_invalid`
  - `package_schema_unsupported`
  - `archive_layout_invalid`
  - `archive_path_traversal_blocked`
- entity contract:
  - `entity_required_field_missing`
  - `entity_shape_invalid`
  - `entity_duplicate_collision`
  - `entity_match_unresolved`
- field and rule enforcement:
  - `protected_field_blocked`
  - `strip_protected_field`
  - `slug_reference_unresolved`
  - `acf_choice_invalid`
  - `site_fingerprint_mismatch`

Validation code strings should remain stable enough that UI rendering and PHPUnit coverage can key off them instead of matching prose message text.

## Translation Rules

The AI submission package should not be imported directly as-is.

Instead, the intake layer should:

1. validate the archive
2. normalize accepted entity JSON
3. map files into standard DBVC sync destinations
4. write translated JSON into a safe staging workspace
5. hand off to the existing importer

Translation should preserve canonical entity contents while resolving:

- sync folder paths
- filename policy
- safe staging directories
- optional import report artifacts

### Translated Artifact Files

Recommended retained translation files:

- `translated/translation-manifest.json`
- `translated/sync-root/{post_type}/{filename}.json`
- `translated/sync-root/taxonomy/{taxonomy}/{filename}.json`

The `translated/sync-root/` subtree should mimic standard DBVC sync layout while remaining isolated inside the intake workspace.

This contract intentionally does not require writing directly into the live sync folder. Import handoff may:

1. call importer helpers against files in `translated/sync-root/`, or
2. copy those files into the real sync folder only if compatibility requires it

### Translation Manifest Shape

Recommended top-level keys:

- `translation_schema_version`
- `intake_id`
- `generated_at`
- `source_package`
- `validation_status`
- `summary`
- `artifacts`

Recommended example:

```json
{
  "translation_schema_version": 1,
  "intake_id": "ai-intake-20260407-120000-abc123",
  "generated_at": "2026-04-07T12:32:00Z",
  "source_package": {
    "package_type": "dbvc_ai_submission_package",
    "package_schema_version": 1,
    "site_fingerprint": "sha256..."
  },
  "validation_status": "valid_with_warnings",
  "summary": {
    "translated_posts": 4,
    "translated_terms": 2,
    "create_entities": 3,
    "update_entities": 3,
    "warnings": 3
  },
  "artifacts": [
    {
      "entity_key": "post:page:about-us",
      "source_path": "entities/posts/page/about-us.json",
      "translated_relative_path": "sync-root/page/page-about-us.json",
      "entity_kind": "post",
      "object_type": "page",
      "operation_intent": "update",
      "resolved_target": {
        "id": 42,
        "vf_object_uid": "vf_post_abc123",
        "slug": "about-us"
      },
      "importer": "DBVC_Sync_Posts::import_post_from_json"
    }
  ]
}
```

### Translation Artifact Record Shape

Each `artifacts[]` record should describe one translated importer-ready file.

Recommended fields:

- `entity_key`
- `source_path`
- `translated_relative_path`
- `entity_kind`
- `object_type`
- `operation_intent`
- `resolved_target`
- `importer`

Notes:

- `translated_relative_path` should be relative to the `translated/` workspace root.
- `importer` should identify the intended downstream import helper:
  - `DBVC_Sync_Posts::import_post_from_json`
  - `DBVC_Sync_Taxonomies::import_term_json_file`

### Translation Output Rules

Translated files must be importer-ready DBVC JSON, not the original AI-authored logical payloads.

#### Posts and CPTs

Translated post/CPT files should:

- live under `translated/sync-root/{post_type}/`
- use the current DBVC filename policy via existing filename helpers
- preserve canonical top-level DBVC post fields
- preserve `post_type`, `post_title`, and `post_name`
- set `ID` to:
  - resolved local post ID for update-intent entities
  - `0` for create-intent entities
- preserve top-level `vf_object_uid` only for resolved update-intent entities
- synthesize storage-ready `meta` values, including ACF underscore reference keys where required
- strip all `strip_with_warning` fields before writing
- never write blocked fields into translated artifacts

#### Terms

Translated term files should:

- live under `translated/sync-root/taxonomy/{taxonomy}/`
- use the current DBVC filename policy via existing filename helpers
- preserve canonical top-level term fields
- set `term_id` to:
  - resolved local term ID for update-intent entities
  - `0` for create-intent entities
- preserve top-level `vf_object_uid` only for resolved update-intent entities
- prefer portable parent linkage that remains compatible with current term import behavior
- strip all `strip_with_warning` fields before writing
- never write blocked fields into translated artifacts

#### References and Relationships

Translated artifacts should resolve AI-facing logical references before import handoff.

Examples:

- structured slug parent refs should become importer-ready `post_parent` or `parent_slug` values
- structured taxonomy refs should become importer-ready `tax_input` items
- ACF relationship-like field values should become storage-ready values aligned to the current site

#### Translation Failure Behavior

If translation encounters a failure after validation reported the package as accepted:

- translation should stop before importer handoff
- the intake should record a blocking translation issue in the retained reports
- the package should be treated as blocked for import purposes
- partial translated artifacts must not be mistaken for import-approved output

## Existing Importer Compatibility

The current importer contract remains authoritative.

The AI package feature should build on existing DBVC behavior already used for:

- post import JSON
- term import JSON
- sync-folder routing
- ZIP staging

The AI package contract should therefore favor importer-compatible entity shapes over a custom abstract format.

- warning handling policy

- allow create-only submissions
- allow create-and-update submissions

### AI Providers and Integrations

- provider name
- API credentials
- model defaults
- service mode

These provider settings are future-facing and optional. They are not required for sample package generation or AI package validation/import.

## Pre-Implementation Decisions

This section captures decisions that should be treated as locked planning defaults unless a later implementation review deliberately changes them.

### Entity Identity and Operation Inference

V1 should support mixed `create_and_update` packages.

Operation intent is inferred per entity:

- `create intent`
  - posts and CPTs use `ID: 0`
  - terms use `term_id: 0`
  - `vf_object_uid` must be empty
- `update intent`
  - a non-empty `vf_object_uid` is present, or
  - a positive `ID` or `term_id` is present

Rules:

- create-intent entities may be created when validation passes
- update-intent entities must resolve to a local match before import
- update matching precedence is:
  1. `vf_object_uid`
  2. slug
  3. numeric ID
- if an update-intent entity cannot be matched, that entity is blocked
- sample package templates for net-new entities should not emit `vf_object_uid`
- submission packages may include `vf_object_uid` only for update-intent entities

### Protected and System-Managed Field Policy

The intake layer should classify protected/system-managed fields into three buckets:

- `allowed`
- `strip_with_warning`
- `blocked`

#### Posts

Strip with warning:

- `meta._edit_lock`
- `meta._edit_last`
- `meta._wp_old_slug`

Blocked:

- `meta._dbvc_import_hash`
- `meta.dbvc_post_history`
- proposal-review decision blobs
- resolver-specific artifacts
- raw proposal manifest metadata
- any attempt to send `vf_object_uid` inside `meta`

Conditionally blocked:

- `meta._thumbnail_id`
  - blocked when populated with numeric IDs in v1
  - may be revisited later when media-aware AI package import exists

Top-level `vf_object_uid` is allowed only for update-intent entities.

#### Terms

Strip with warning:

- `meta._edit_lock`
- `meta._edit_last`

Blocked:

- term history/import bookkeeping blobs
- any attempt to send `vf_object_uid` inside `meta`

Top-level `vf_object_uid` is allowed only for update-intent entities.

### ACF Discovery and Coverage Boundaries

V1 should use a layered ACF discovery strategy:

1. local ACF JSON when available
2. runtime ACF APIs when available
3. registered meta and DBVC-safe fallbacks

Local ACF JSON is a first-class discovery source because it exposes:

- field group keys
- field names and labels
- field types
- field choices
- `location` rules
- nested `sub_fields`
- `layouts`
- clone relationships

This is viable for current Vertical Framework sites using paths such as:

- `/wp-content/themes/vertical/acf-json`

The generator should support additional detected `acf-json` roots later, but the implementation guide should treat theme-level local JSON as a first-class source from day one when present.

#### First-Class ACF Field Families In V1

First-class storage-aware translation support:

- scalar/textual fields:
  - `text`
  - `textarea`
  - `wysiwyg`
  - `email`
  - `url`
- scalar/choice fields:
  - `number`
  - `range`
  - `true_false`
  - `select`
  - `checkbox`
  - `radio`
  - `button_group`
- date/time style fields:
  - `date_picker`
  - `date_time_picker`
  - `time_picker`
- reference-style fields:
  - `post_object`
  - `relationship`
  - `taxonomy`

First-pass storage-aware structural translation support for supported leaf families:

- structural fields:
  - `group`
  - `repeater`
  - `flexible_content`
  - `clone`

Additional v1 note:

- nested scalar/reference descendants within these structures may be translated when the local field catalog can resolve them safely to the expected meta names
- first-pass clone support is storage-aware but still requires broader regression coverage for advanced clone combinations
- `page_link` remains deferred until its storage behavior is handled explicitly

Document-only or non-authoring field types should not be treated as importable content fields:

- `tab`
- `accordion`
- `message`

Deferred or restricted in v1:

- `image`
- `file`
- `gallery`
- other media-centric fields requiring media asset packaging or attachment ID resolution

Non-empty values for unsupported complex or deferred media-centric ACF families should be blocked in v1 rather than guessed or silently stripped.

#### ACF Logical Representation Strategy

The AI-facing contract should use logical field names and logical structures, not raw ACF storage internals.

Recommended representations:

- `group`
  - object keyed by subfield name
- `repeater`
  - array of row objects keyed by subfield name
- `flexible_content`
  - array of block objects using `acf_fc_layout` plus layout field keys
- `clone`
  - expanded inline using the cloned field structure while docs note the clone source

The intake layer should:

- map field names to field keys using local JSON or ACF API data
- synthesize underscore reference meta such as `_field_name => field_abc123`
- translate logical nested ACF structures into import-ready `meta` payloads

### Relationship and Reference Policy

V1 should be slug-based, not ID-based, for AI-authored references.

The canonical authoring format should be structured slug references rather than bare numeric IDs.

Recommended formats:

- post relationship-like fields:
  - `{ "slug": "about-us", "post_type": "page" }`
- taxonomy relationship-like fields:
  - `{ "slug": "consulting", "taxonomy": "service_type" }`
- parent post references:
  - `{ "slug": "parent-page", "post_type": "page" }`

Plain slug strings may be tolerated only where the target context is already unambiguous, but the canonical format should remain structured slug references.

This is possible in v1 because the intake layer can resolve slugs against the current site's posts, CPTs, and taxonomy terms before translation into importer-ready values.

For newly created entities inside the same AI package, v1 should support a second-pass post-import resolution layer so parent links, taxonomy assignments, and supported ACF relationship values can be backfilled using the final local IDs created during import.

### Warning vs Blocked Matrix

The implementation guide should expand this into a full rule table, but the following defaults are locked now.

Warning:

- advisory markdown rule not followed
- optional field omitted
- extra non-contract docs included in the archive
- strip-eligible protected fields encountered
- unknown optional metadata that can be safely discarded
- unsupported but empty optional fields

Blocked:

- missing or invalid manifest
- unsupported package type
- unsupported package schema version
- malformed JSON
- duplicate entity collisions inside the package
- unknown post type
- unknown taxonomy
- missing required field
- invalid field type/shape for canonical contract
- unsupported non-empty complex ACF value
- unsupported non-empty deferred media ACF value
- forbidden protected/system-managed field usage
- unresolved update-intent entity match
- unresolved required slug reference
- invalid ACF choice value when the rule is machine-enforced
- site fingerprint mismatch

`valid_with_warnings` should be supported, but blocked issues must prevent staging translation and importer handoff.

### User-Authored Rules Schema

Markdown guidance is necessary but not sufficient. V1 should also support a machine-readable rules object with defaults applied in the admin inputs.

First-pass structured schema:

```json
{
  "schema_version": 1,
  "defaults": {
    "package_mode": "create_and_update",
    "post_status": "draft",
    "warning_policy": "confirm",
    "value_style": "blank"
  },
  "global": {
    "notes_markdown": "",
    "required_post_fields": ["post_title", "post_name"],
    "blocked_meta_keys": [],
    "blocked_field_paths": [],
    "allowed_statuses": ["draft", "publish"],
    "validation": {
      "unknown_extra_fields": "warn",
      "invalid_choice_values": "block"
    }
  },
  "post_types": {
    "page": {
      "notes_markdown": "",
      "defaults": {
        "post_status": "draft"
      },
      "required_taxonomies": [],
      "required_meta_keys": [],
      "required_acf_fields": [],
      "field_rules": {
        "meta.hero_title": {
          "required": false,
          "enforcement": "warn"
        }
      }
    }
  },
  "taxonomies": {
    "category": {
      "notes_markdown": "",
      "allow_create": true,
      "required_fields": ["name", "slug"]
    }
  }
}
```

Rule inputs in the UI should ship with defaults pre-populated rather than starting fully blank.

### Site Fingerprint Definition

V1 should define `site_fingerprint` as a schema-oriented deterministic hash, not a content hash.

Recommended inputs:

- supported post types and supports
- attached taxonomies and taxonomy object mappings
- registered meta catalog signature
- ACF field catalog signature
- user-authored rules signature
- AI package schema version

Manifest metadata should separately record `site_origin` details such as:

- `home_url`
- `site_name`
- `blog_id` when available

Those origin values should be informative metadata rather than the primary compatibility hash.

### Submission Package Version Tolerance

V1 should require an exact supported package schema version match.

That means:

- exact integer version match only
- no forward minor-version tolerance in v1
- semver-style compatibility can be introduced later when the package contract stabilizes further

### Intake Staging, Cleanup, and Logging

Recommended intake workspace:

- `wp-content/uploads/dbvc/ai-intake/{intake_id}/`

Recommended subdirectories:

- `source/`
- `extracted/`
- `translated/`
- `reports/`

Logging:

- use the existing DBVC logging system
- log package detection, validation summary, translation summary, and import handoff results

Retention and cleanup:

- keep full dry-run artifacts
- keep validation reports for blocked packages
- keep successful intake workspaces long enough for debugging and operator review
- first-pass default retention should be time-based rather than immediate deletion

Recommended default:

- retain intake workspaces for 7 days
- clean them with a scheduled cleanup task later
- allow future configurability if disk usage becomes a concern

## Summary

This feature should be built around a strict but practical rule:

- AI sees canonical DBVC-shaped templates plus strong docs.
- DBVC receives a stricter AI submission package, validates it, translates it, and then reuses the existing importer.

That keeps the authoring workflow AI-friendly while preserving DBVC's current import architecture.
